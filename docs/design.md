# Baroboys — Design Philosophy

A compressed mental-model guide for anyone (AI or human) new to the repo.
For command references see `CLAUDE.md`; for implementation detail see `docs/architecture.md`.

---

## The Four Mental Models

### 1. Packer builds the machine; Git runs it

Packer bakes a complete, self-contained GCE image — OS, packages, systemd units, Wine prefix,
game binaries. It runs **once per dependency change**.

Everything else deploys through Git. The VM pulls `origin/main` on every boot. Changing a bash
script or config template just needs a commit and push — the change is live at next reboot. Only
changing system packages, unit file enable/disable state, or compiled assets requires a new image.

### 2. Git is the source of truth — including save state

The VM treats its working tree as **tainted**. Local state accumulates (envsubst'd configs,
SteamCMD artifacts, in-progress writes). The shutdown sequence deliberately stashes this taint,
rebases against origin, pushes, then restores. The VM only trusts what's in `origin/main`.

The repo also serves as the game-state database. Each clean shutdown commits the latest
compressed save and pushes. This gives version history and simple restore at the cost of binary
bloat over time (managed periodically by `make clean-git`).

### 3. systemd owns all lifecycle

Boot ordering, service readiness, shutdown sequencing — all expressed in systemd, not shell.
Shell scripts are the *implementation*; systemd is the *orchestrator*.

Concretely:
- Service ordering: `After=` + `Requires=` (pairing both is mandatory — `Requires=` alone does
  not enforce order)
- Process-level readiness: `ExecStartPost=` — e.g. Xvfb polls for its own socket before
  reporting active, so `systemctl start xvfb-startup.service` is safe to call and block on
- Shutdown hooks: `game-shutdown.service` is wired into `poweroff/halt/reboot` targets via
  `[Install]` — runs on **any** stop event, not just idle-check
- When you encounter a sequencing or readiness problem, reach for a systemd mechanism first.
  Sleeps and polling loops belong in `ExecStartPost=` if anywhere — not in provisioner scripts.

### 4. Two-phase service pattern

Every component has two units: `*-setup.service` (oneshot, root, installs/configures) followed by
`*-startup.service` (long-running or oneshot, `bwinter_sc81`, runs the process). This is a
**security boundary** — root does system-level setup once, then hands off to an unprivileged user
for the long-running process. See `docs/architecture.md` for the full dependency graph.

## Cost Philosophy

**Optimize for near-zero idle cost.** The server runs only during active play sessions; it is
fully destroyed between sessions (not just stopped — `make destroy` tears down the VM entirely).
This is a hard constraint: any new feature or infrastructure addition should be evaluated against
its always-on cost. A stopped VM costs nothing; a Cloud Function costs near-nothing but is still
"always on" in billing terms; a second persistent VM is a non-starter.

Consequences:
- Single VM for everything (game + admin panel + idle-check) — no separate management server
- Admin panel is on the game VM, so it can't help start a stopped VM (the circular dependency is
  intentional — fixing it would require always-on infrastructure)
- `make start` from a local terminal is the correct start mechanism; no bookmarkable URL is
  worth adding always-on infra to support

---

## Key Tradeoffs

| Decision | Benefit | Cost |
|----------|---------|------|
| Git as save database | Simple backup, version history, no extra infra | Binary history grows; needs periodic BFG cleanup (`make clean-git`) |
| Full VM destroy between sessions | Near-zero idle cost | Reboot takes ~5 min boot sequence |
| Wine for VRising | Run Windows-only game on Linux VM | Wine prefix must be initialised at Packer build time; wineboot failures break the image build |
| systemd-only lifecycle | Clean, composable, testable | Unit changes require a full image rebuild to take effect |
| `.in` template pattern | Secrets never committed; config regenerated fresh on each boot | Runtime config in `StreamingAssets/Settings/` is gitignored; always regenerated by `envsubst` |

## Gotchas for New Contributors

**Unit changes require a new Packer image.**
Units are baked in at build time via `setup.sh` calling `systemctl enable`. A unit change that
isn't followed by `make build-game-vrising` (or the relevant game) won't take effect.

**The shutdown stash is load-bearing.**
`git stash push → pull --rebase → push → stash pop` in `shutdown.sh` intentionally clears
working-tree taint before rebasing. Do not simplify this to a bare `git fetch && git rebase`.
The stash step is what makes the rebase reliable in a dirty working tree.

**The warm SteamCMD call is intentional.**
`refresh.sh` runs a bare `+login anonymous +quit` before the real `app_update`. This works around
intermittent SteamCMD installer failures (likely depot cache initialisation). Root cause unknown;
removing it makes builds flaky. The comment in the file explains this.

**Wine prefix is initialised at build time, not runtime.**
`wineboot` + `winetricks` run during Packer's VRising image build. The prefix lives in the baked
image. At VM boot, Wine just uses the pre-built prefix in `~/.wine64`.

**`Type=simple` does not mean "ready".**
`systemctl start` returns as soon as the process forks — before it accepts connections. Use
`ExecStartPost=` to express readiness. `xvfb-startup.service` is the canonical example: it polls
for `/tmp/.X11-unix/X0` before reporting active.

**`wine64` was removed in Wine 11 (Jan 2026).**
The unified `wine` binary handles both 32-bit and 64-bit PE binaries. All invocations use
`/opt/wine-stable/bin/wine`. `WINEARCH=win64` still works as expected.
